## 0. v2 → v3 の主な変更

1. **設定を最小化**（§3.6）。美的パラメータ（背景／雨量／演出強度）を設定から外し、作者が決め打ちする。設定に残すのは機能（サイクル長）と基本操作（音量・ミュート・自動起動）の3点のみ。
2. **差別化を明文化・確定**（§2.5）。対Junの楔を2本に固定 ── (A) 予兆フェーズ（休憩への滑らかな移行の発明）、(B) 集中中の常時・数字なしバー（隅のクリックスルーHUD、既定ON）。"オプション"表記を廃止。
3. **時間表示の方針を確定**。集中＝穏やかな連続バー（数字なし）、休憩＝環境表現（雨）。状態ごとに見せ方を変えるのを設計判断として固定する。
4. **最優先の技術検証ゲートを新設**（§4・§11）。macOSでユーザーのネイティブ全画面アプリの上にオーバーレイが出るか。ここが転ぶと中核体験が成立しないため、着工前に潰す。
5. **配布の事実を更新**（§8）。Azure Trusted Signing は「**Azure Artifact Signing**」に改称、日本（個人・法人）は現在も対象外であることを明記。WebView2 は「大多数に配布済み・ごく一部は未導入」を bootstrapper で吸収、と精度を補正。
6. **「雨の作業モード」を roadmap から除外**。集中中の常時降雨は予兆ランプの可読性と排他になるため、採らない。

## 1. プロジェクト概要

雨やどり（仮称）は、ポモドーロ型の作業タイマーと、窓ガラスを流れる雨のアンビエント表現を組み合わせたデスクトップ常駐アプリである。集中中は隅に置いた穏やかなバー（数字なし）だけが時間の経過を静かに伝え、画面の大半は何も出さない。休憩が近づくと透過した雨が画面に重なって徐々に強まり、休憩本体では雨が覆って休息を促す。雨が上がれば静かに引っ込み、次の作業へ戻る。

「通り雨が来たから少し手を止める」という体験を、強制ではなく誘いとして設計する。梅雨どきのしんみりした空気感を基調に、集中と休息のリズムをやわらかく区切ることを目的とする。

| 項目 | 内容 |
|---|---|
| 対応プラットフォーム | macOS ＋ Windows（Linux は対象外） |
| 利用シーン | 在宅・個人作業中の集中／休憩サイクル管理 |
| 開発体制 | 個人開発 |
| 配布 | GitHub Releases（未署名、インストール手順を明記） |

## 2. コンセプト / 世界観

- **テーマ**: 梅雨の通り雨。休憩＝雨やどり。
- **トーン**: 静か、しんみり、急かさない。
- **中核体験**: 作業（隅に穏やかなバー）→（透過の雨が30秒かけて近づく予兆）→ 通り雨で休息 → 雨上がり → 作業、の反復。
- **設計原則**: 休息は強制ではなく誘い。逃げ場（Skip／中断）を必ず残す。

### 2.5 差別化方針（対 Jun ほか）

Jun（macOS専用、時間を一切表示せず、休憩はブラーで受動的に促す）に対し、雨やどりの楔は2本に固定する。

- **(A) 予兆という「移行」の発明。** 休憩はいきなり来ない。休憩30秒前から雨が近づき、作業を続けたまま「もうすぐ通り雨」を体で察知できる。Jun の「いきなり降る」とも、Stretchly 等の強制中断とも違う、唯一の移行設計。
- **(B) 集中中の穏やかな常時バー。** Jun が時間を隠すのに対し、雨やどりは「見せる、ただし穏やかに」。隅に数字なしの連続バーを常駐させ、ゴールが近づく感覚（goal-gradient）で集中を後押しする。表示専用のため操作を妨げない。

この2本は層が異なる（バー＝集中中の静かな時間提示、予兆／通り雨＝休憩への episodic な移行）ため競合せず、互いを補強する。

> *代替案*: 「集中中も画面に何も出さない」純ミニマル路線（Jun 寄り）を採るなら、HUD を廃してトレイ表示のみにできる。ただしその場合、差別化は (A) 予兆の一点に依存する。本書は **(A)+(B) 併用を既定**とし、純ミニマルはビルド時の分岐として残す（ユーザー設定にはしない）。

## 3. 機能要件

### 3.1 タイマー / フェーズ管理
- 作業時間・休憩時間を持つサイクルタイマー（デフォルト：作業20分 / 休憩5分）。時間は設定で可変（§3.6）。
- 状態機械として以下の4フェーズを持つ：
  1. **作業（Work）** — 通常作業中。全画面オーバーレイは退避し、隅の HUD バーのみ表示。
  2. **予兆（Incoming）** — 休憩30秒前から。全画面の透過オーバーレイをクリックスルーで重ね、雨を 0→強 へ漸増。**作業は継続できる**。HUD バーは作業時間の残り（ほぼ満了）を示す。
  3. **通り雨（Shower）** — 休憩本体。クリックスルーを解除し、雨が画面を覆う。**Skip 等が押せる**。HUD バーは非表示（休憩は環境表現のみ）。
  4. **雨上がり（Clearing）** — 雨が弱まりフェードアウトして退避。作業復帰時に HUD バーが再表示。
- 雨上がり完了後、作業へ戻りループする。
- 残り時間の把握手段は、集中中＝隅の HUD バー、いつでも＝トレイ／メニューバー表示の二段で提供する。

### 3.2 ウィンドウ制御（フェーズ連動）
本アプリは2つのウィンドウを持つ。**(1) 隅の HUD ウィンドウ**（小型・最前面・常時クリックスルー・表示専用）と、**(2) 全画面の雨オーバーレイ**。

- **作業**: 雨オーバーレイは `hide()`。HUD のみ表示（バー）。他アプリ操作を一切妨げない。
- **予兆**: 雨オーバーレイを `show` → 全画面・透過・最前面 → `setIgnoreCursorEvents(true)`（クリックスルーON）。雨の濃度／生成量を tween で漸増。クリックは後ろのアプリに抜け、作業を継続できる。HUD は表示継続。
- **通り雨**: 雨オーバーレイを `setIgnoreCursorEvents(false)`（クリックスルーOFF）に切替。ウィンドウ全体が操作を受けるようになり、Skip ボタンが機能する。雨は最大に、必要なら濃度／背景を強めて覆う。HUD は非表示。
- **雨上がり**: 雨をフェードアウト → 最前面解除 → `hide`。HUD を再表示。
- 全画面オーバーレイの always-on-top がONなのは予兆〜雨上がりの間のみ。HUD は全フェーズで最前面・クリックスルー。

### 3.3 雨レンダリング
- 窓ガラスに付いた雨粒が背景を屈折させ、合流し、尾を引いて流れる表現。
- 背景は **ぼかした風景の静止画1枚**を屈折元に使う（モードA、決め打ち。設定では切り替えない）。
- 雨脚をフェーズ進行に合わせ変化（予兆＝0→強の漸増、通り雨＝強、雨上がり＝強→消滅）。

### 3.4 音
- 雨音を休憩開始でフェードイン、終了でフェードアウト。音量設定・ミュートを提供する。

### 3.5 休憩中の操作 / 逃げ道
- 「この通り雨をやり過ごす（Skip）」を画面に配置（通り雨フェーズでクリックスルーOFFのため機能する）。
- Esc などで休憩を中断して作業に戻れる。
- 休憩を“逃げ場なし”にしない。クリックスルー無効化による操作ロックは**行わない**（覆うのは視覚のみ、Skip/Esc は常に有効）。

### 3.6 設定（最小）
設定はユーザーに「道具を管理させる」負債であり、アンビエントの価値を削る。美的パラメータは作者が1つに決め打ちし、設定から外す。残すのは以下のみ。

- **作業時間 / 休憩時間**（デフォルト 20分 / 5分）── 唯一の機能的可変項目。作業リズムは人により大きく異なる（25/5 と 90分集中では別物）ため、ここだけ可変を残す。
- **音量 / ミュート** ── 好み設定ではなく基本操作。
- **起動時の自動開始 ON/OFF** ── 常駐アプリの標準的利便。

設定値は永続化する。**背景・雨量・演出強度はユーザー設定にしない**（作者が決定）。`prefers-reduced-motion` は設定ではなく自動で尊重する（§4）。

### 3.7 トレイ / メニューバー
- 常駐アイコン。現在フェーズ・残り時間を表示。開始 / 一時停止 / Skip / 設定 / 終了の操作を提供する。

## 4. 非機能要件

| 区分 | 要件 |
|---|---|
| パフォーマンス | FPS上限を設定（30fps目安）。待機・非表示・被覆時はレンダリング停止し省電力。**内蔵GPU機で通り雨フルスクリーン時の実機FPSを計測**し、未達なら雨量／屈折を落とす（§11） |
| 常駐性 | 長時間起動を前提とし、バッテリー消費を抑える（軽量フットプリント重視） |
| 対応OS | macOS（WKWebView）／Windows（WebView2）。両者とも WebGL 実用上問題なし。Linux は対象外 |
| **macOS全画面共存** | ユーザーのネイティブ全画面アプリ（全画面エディタ／動画／Zoom 等）の**上に**予兆・通り雨を表示できること。`NSWindowCollectionBehaviorFullScreenAuxiliary` ＋ `canJoinAllSpaces` と適切なウィンドウレベルで実現する。**未達なら中核体験が発火しないため最優先の検証対象**（§11） |
| アクセシビリティ | `prefers-reduced-motion` を尊重（演出を簡略化／無効化）。ユーザーを閉じ込めない（Skip/Esc 常備） |
| タイマー精度 | 非表示時もズレない（バックエンド＝Rust側駆動） |

## 5. 技術選定

### 5.1 採用スタック

| レイヤ | 採用 |
|---|---|
| シェル | **Tauri 2** |
| フロントエンド | **SvelteKit** |
| 雨描画 | **WebGL**（raindrop-fx 等の既製ライブラリ、または Codrops 系の自前実装） |
| スケジューラ | **Rust**（tokio interval 等）でフェーズ管理しイベント発火 |
| 音 | Web Audio（Tone.js 等） |

### 5.2 フレームワーク選定理由（なぜ Tauri 2 か）

- **決め手＝常駐フットプリント**: 本アプリはトレイに一日中常駐する。Electron は Chromium+Node 同梱でインストーラ80〜150MB・メモリ150〜300MB、Tauri はシステム標準 WebView＋Rust コアでインストーラ10MB未満・メモリ30〜50MB。アンビエント常駐アプリにとって Electron の常駐コストは最も払いたくないコスト。
- **WebGL の一貫性は両対象OSで問題なし**: Tauri は各OSの WebView に依存するが、Windows の WebView2 は Chromium ベース、macOS の WKWebView も実用上問題なし。弱点になりがちなのは Linux の WebKitGTK のみで、それは対象外。よって両対象OSは Tauri の強いゾーンに収まる。
- **タイマーのバックエンド駆動と相性**: 非表示 WebView の JS タイマーはスロットリングされ周期がズレるため、時間管理を Rust 側に置く設計が前提。これが Tauri の Rust + tokio に素直にハマる。
- **スタック適合**: SvelteKit はそのまま動作。増える Rust はタイマー＋ウィンドウ制御程度で表面積は小さい。

**Electron を選ぶ条件**（不採用の理由の裏返し）: Linux を含む全OSで WebGL を盤石にしたい、または Rust を一切入れたくない場合のみ。本件はどちらにも当たらない。

**検討した代替案**:
- *ネイティブ（Swift + AppKit + Metal）*: macOS専用なら最軽量・最高制御。ただし Windows 対象化により不適（クロスプラットフォーム不可、雨を Metal で書き直し）。
- *Flutter*: 独自 Skia で全OS描画一貫だが、Dart 習得と WebGL 資産の不再利用が代償。
- *Wails v3（Go）*: Tauri の Go 版だが v3 はアルファ段階、かつ Go 不使用のため優先理由薄い。

### 5.3 雨描画の方針
- モードA（自前背景・静止画1枚）を主軸。背景テクスチャを屈折元に使うため `raindrop-fx`（WebGL2、屈折・ミスト・生成レート設定済み）がそのまま使える。
- 雨エフェクトはフレームワーク非依存モジュールとして実装し、将来のブラウザ版とコア共有できる形にする。

### 5.4 クリックスルーの実装方針
- 全画面オーバーレイは**ウィンドウ単位トグル**（`setIgnoreCursorEvents` ON/OFF）で実現。予兆＝ON、通り雨＝OFF。
- 隅の HUD（バー）は**表示専用・常時クリックスルー**とし、クリックを受けない。これにより**ピクセル単位のヒットテストは一切不要**になる（Tauri は per-region hit-test を未提供だが、本設計はトグルのみで成立する）。
- 補足: macOS は透過ウィンドウの完全透明ピクセルが自動で背面に抜けるため、予兆フェーズのクリックスルーは素直に効く。

### 5.5 Windows 固有の留意点
- **WebView2 ランタイム**: Tauri は同梱しない。WebView2 は Windows 11 全機にプリインストール、対象の Windows 10（1803以降）にも配布済みだが、**ごく一部の端末は未導入**。Tauri のインストーラは未導入時に既定で `downloadBootstrapper` により導入するため、その差は吸収でき、既定のまま軽量を維持できる。
- **透過**: alpha 時代にあった Windows 透過不具合は 2.0 安定版で修正済み。透過＋フレームレス＋ドラッグ系に細かな描画不具合が残るため該当箇所は要テストだが、本設計の中核（不透明寄りの全画面・ドラッグなし）には当たらない。
- **実機確認**: macOS/Windows とも、ビルド後の実機で透過挙動を確認する（macOS はバンドル後に透過が外れる既知 issue があるため `macOSPrivateApi: true` 等を確認）。

## 6. アーキテクチャ

### バックエンド（Rust / Tauri Core）
- フェーズの状態機械とタイマー（**時間の真実はここ**）。
- ウィンドウ制御（HUD と全画面オーバーレイの show / hide / 全画面 / 最前面 / フォーカス / クリックスルー切替）。
- トレイ管理。
- フロントへフェーズ遷移イベントを emit。

### フロントエンド（SvelteKit / WebView）
- イベントを受けて雨描画・HUD バー・音・UI を反映。描画と演出に専念し、時間管理は持たない。

### 設計理由
非表示 WebView の JS タイマーはスロットリングで周期がズレるため、時間管理を Rust 側に置く。

## 7. 状態遷移とウィンドウ属性

| フェーズ | 全画面オーバーレイ | 隅HUD（バー） | always-on-top | クリックスルー（オーバーレイ） | 雨 | 音 | 操作 |
|---|---|---|---|---|---|---|---|
| 作業 | hide | 表示（残り＝作業バー） | HUDのみON | — | なし | OFF | — |
| 予兆（30秒） | 全画面・透過 | 表示（バーほぼ満了） | ON | **ON** | 0→強へ漸増 | (任意)小 | 後ろのアプリ操作可（作業継続） |
| 通り雨（休憩） | 全画面 | 非表示 | ON | **OFF** | 強（必要なら濃度up） | フェードイン | **Skip/Esc で作業へ** |
| 雨上がり | 全画面→hide | 作業復帰時に再表示 | ON→OFF | OFF | 強→消滅 | フェードアウト | — |

> HUD は独立した小型・最前面・常時クリックスルーのウィンドウ。表示専用のためクリックを受けず、ヒットテスト不要。

**遷移**: 作業 →(20分経過・残り30秒)→ 予兆 →(作業終了)→ 通り雨 →(5分経過)→ 雨上がり →(完了)→ 作業 …（ループ）。Skip／中断はどのフェーズからでも作業へ戻せる。

## 8. 配布

### 方式
- **GitHub Releases ＋ `tauri-apps/tauri-action`**。アクションが macOS(Arm/Intel)・Windows のインストーラをビルドし、Release の作成＆アップロードまで自動化。バージョンタグ（`v*`）の push で発火。
- 生成物: macOS `.dmg`、Windows `.msi`(WiX) と `-setup.exe`(NSIS)。
- **未署名で配布**し、README とリリースノートにインストール手順を明記する。
- 自動アップデート（任意）: updater プラグイン＋ `includeUpdaterJson: true` で `latest.json` を Release に同梱。Tauri 更新用署名鍵（`TAURI_SIGNING_PRIVATE_KEY`、OSのコード署名とは別物）が必要。

### インストール手順（配布ページ掲載用ドラフト）

**macOS**
1. `.dmg` を開き、アプリを「アプリケーション」へドラッグ。
2. 初回のみ: アプリを右クリック →「開く」→ ダイアログで「開く」（未署名のため。2回目以降は通常起動）。
3. 「壊れているため開けません」と出る場合: ターミナルで
   `xattr -dr com.apple.quarantine /Applications/雨やどり.app`

**Windows**
1. `-setup.exe` を実行。
2. 「Windows によって PC が保護されました」が出たら →「詳細情報」→「実行」（未署名のため）。

### 将来のコード署名（任意・段階的）
- **macOS**: Apple Developer Program（年99ドル）→ Developer ID 署名＋notarization で Gatekeeper 警告を解消。
- **Windows**: **Azure Artifact Signing（旧 Azure Trusted Signing、月約9.99ドル/5,000署名）は、日本の個人・法人とも現在も対象外**。公開信頼証明書の提供は米・加の組織／個人、および EU・英の組織のみ。なお制限がかかるのは「検証される組織」の所在地であり、申請するテナントの場所ではないため、米/加法人を検証する逃げ道は理論上あるが個人には非現実的。日本での現実解は従来型 OV 証明書（年2〜4万円前後＋鍵の HSM/トークン保管）。EV の SmartScreen 即時回避特典は 2024 年に廃止済みで、署名しても新規ファイルは評価が貯まるまで警告が出る点に留意。
- ポートフォリオ用途では未署名＋手順明記で割り切り、必要が出たら段階的に署名を追加する。

## 9. MVP スコープと段階開発

### MVP（最小）
- 設定可能な 20/5 サイクル。
- フェーズ：作業（オーバーレイ hide ＋ HUD バー）／予兆（全画面・透過・クリックスルーON・雨漸増・HUD 表示）／通り雨（クリックスルーOFF・Skip 可・HUD 非表示）／雨上がり（退避）。
- 雨描画（自前背景1枚 ＋ raindrop-fx）。
- 隅の HUD バー（数字なし・常時クリックスルー）。
- トレイ常駐 ＋ 残り時間表示。
- Skip／Esc。
- 設定は最小（サイクル長・音量/ミュート・自動起動）。

## 10. 未確定事項 / 要検討

決定済みは反映済み（背景＝静止画1枚、設定＝最小、割り込み＝MVPは固定タイマー）。残る論点は以下。

- **全画面化の方式**: OSフルスクリーン vs 装飾なしの maximize（柔らかさ・オーバーレイ挙動の素直さを優先するなら後者が有力。要実機確認）。

## 11. 着工前の技術検証ゲート

本仕様の前提を支える2点を、フル実装の前に最小プロトタイプで確認する。ここが転ぶと設計を組み替える必要がある。

1. **macOS：ユーザーの全画面アプリ上へのオーバーレイ表示。**
   全画面エディタ／動画／Zoom 上で、予兆の透過オーバーレイと通り雨が実際に前面に出るか。`collectionBehavior`（`FullScreenAuxiliary` ＋ `canJoinAllSpaces`）と window level の組み合わせで解決できるか。
   *解決不能の場合のフォールバック*: 「全画面アプリ中は予兆をスキップし、トレイ通知に降格」等を設計へ追加する。

2. **予兆 → 通り雨のクリックスルー切替の体感。**
   `setIgnoreCursorEvents` の ON→OFF 切替時に、クリックの取りこぼし・誤クリック（切替の瞬間に背後アプリへ抜ける／抜けない）がないか、macOS / Windows 両方で確認。

> 副次的に、内蔵GPU 機での通り雨フルスクリーン時の実機FPS（§4）も同じプロトで測る。
