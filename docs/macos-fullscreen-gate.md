# macOS 全画面共存ゲート — 実機検証手順

> 要件 §11 / 実装計画 Phase 0 の **Gate 1（全画面共存）・Gate 2（クリックスルー切替）** を macOS 実機で合否判定するためのチェックリスト。
> ネイティブ実装は本実装に組み込み済み（`src-tauri/src/macos.rs`）。**この文書の手順で合格を確認するまで、ゲートは「未了」のまま**。

---

## 1. 何を実装したか（＝検証対象）

| 実装 | 場所 | ねらい |
|---|---|---|
| `collectionBehavior = canJoinAllSpaces + fullScreenAuxiliary + stationary + ignoresCycle` | `macos.rs`（起動時、overlay / HUD 両方） | 全 Space に参加し、**他アプリのネイティブ全画面 Space の上**にも表示できる補助ウィンドウにする。Mission Control の整列・ウィンドウ巡回からも外す |
| `sharingType = none` | `macos.rs`（同上） | 自窓を画面キャプチャから除外。モードB（`capture_screen`）の屈折元に雨・HUD 自身が写り込む多重露光を防ぐ。Windows の `SetWindowDisplayAffinity` と対 |
| overlay の `set_focusable(false)` | `windows.rs` init | tao の `show()` は makeKeyAndOrderFront でキーフォーカスを奪うため、`canBecomeKeyWindow` を切って**フォーカスを奪わず**前面表示させる（予兆中の「作業は継続できる」を守る） |
| overlay をモニタ全域へ明示フレーム | `windows.rs` `cover_primary_monitor` | 装飾なし窓の `maximize()` は visibleFrame（メニューバー・Dock を除く）止まりで雨に隙間が出るため |

**window level は Floating(3) のまま**（設計判断）: メニューバー(24) やポップアップ(101) より上へ引き上げると、通り雨中にトレイメニューや Zoom の操作 UI まで覆い「逃げ場を残す」原則に反する。全画面 Space 上のアプリ内容は通常 level(0) なので Floating で覆える想定。実機で覆えないものが見つかった場合は §5 を参照。

## 2. 事前準備

1. 設定（トレイ → 設定…）で **作業 1 分 / 休憩 1 分**にしてサイクルを速く回す（検証後に戻す）。
2. OS の「視差効果を減らす」をオフにする（オンだと雨が出ず静的ベールになる → README トラブルシューティング参照）。
3. `pnpm tauri dev` でも検証できるが、最終判定は **ビルド済み `.app`**（`pnpm tauri build`）で行う（`macOSPrivateApi` 等のバンドル差を含めるため）。

## 3. チェックリスト — Gate 1: 全画面共存

| # | シナリオ | 手順 | 期待結果 | 結果 |
|---|---|---|---|---|
| 1 | 通常デスクトップ（回帰） | 何も全画面にせず 1 サイクル回す | 予兆→通り雨→雨上がりが従来どおり。雨が画面の縁（メニューバー・Dock の裏）まで届き隙間がない。メニューバー・Dock 自体が雨の上に見えるのは仕様 | ☐ |
| 2 | 全画面動画 | Safari/Chrome で動画を**ネイティブ全画面**にして待つ | 予兆・通り雨が**動画の上に**重なる。Space が切り替わらない・全画面が解除されない・再生が止まらない | ☐ |
| 3 | 全画面エディタ（フォーカス非奪取） | VS Code 等を全画面にし、**予兆中もタイピングを続ける** | 1 文字も落ちない（フォーカスを奪われない）。クリックも背後のエディタに抜ける | ☐ |
| 4 | 通り雨からの逃げ場 | 全画面アプリ上で通り雨に入る | 「やり過ごす（Skip）」が**最初の 1 クリック**で効く。Esc でも抜けられ、元の全画面アプリにそのまま戻れる | ☐ |
| 5 | Zoom（会議・画面共有） | Zoom 会議を全画面＋画面共有して待つ | 自分には予兆・通り雨が見える。**共有相手の画面に雨が写らない**（sharingType）。Zoom のミュート等の操作 UI は雨より上に出て操作できる | ☐ |
| 6 | Mission Control / Cmd+Tab | 通り雨中に Mission Control・Cmd+Tab | overlay がウィンドウ一覧に並ばない・巡回に出ない（stationary / ignoresCycle） | ☐ |
| 7 | Space 横断 | 通常 Space を 2〜3 枚作り行き来する | HUD がどの Space でも同じ位置に見える | ☐ |
| 8 | モードB 多重露光 | 通り雨を 2〜3 回繰り返す | 雨ガラスの屈折元（背景）に前回の雨や HUD が写り込んでいない | ☐ |

## 4. チェックリスト — Gate 2: クリックスルー切替の体感

| # | 手順 | 期待結果 | 結果 |
|---|---|---|---|
| 1 | 予兆の最後の数秒、背後のエディタをクリック連打 | すべて背後に届く（オーバーレイに吸われない） | ☐ |
| 2 | 通り雨に変わった直後、すぐクリック | オーバーレイ（Skip 等）に届き、背後アプリに抜けない。境界の 1 クリックの取りこぼしまでは許容（実装計画 §2 の合格基準） | ☐ |

> 副次（Gate 3）: 同じ実機で通り雨フルスクリーン時の FPS をメモる（内蔵 GPU 機・30fps 目安）。

## 5. 不合格だった場合の分岐

- **全画面 Space 上に何も出ない（Gate 1 否）** → 要件 §11 のフォールバック「全画面アプリ検知中は予兆をスキップし、トレイ通知へ降格」へ設計を組み替える。
- **全画面アプリの一部 UI が雨より上に残る／覆えない** → window level の引き上げを検討する。`macos.rs` の `with_ns_window` 内で `ns_window.setLevel(objc2_app_kit::NSStatusWindowLevel)`（25）等を試す一行で済む。ただし **101（ポップアップ）以上に上げるとトレイメニューや Zoom 操作 UI まで覆って逃げ場を塞ぐ**ため、上げ幅は最小にする。
- **通り雨で Skip の 1 クリック目が効かない**（非キー窓へのクリック配送の問題）→ overlay の `set_focusable(false)`（`windows.rs` init）を、通り雨進入時だけ `true` へ切り替える方式を検討（予兆では必ず false を維持）。
